Phase A — Stronger creator inputs

Phase C — Semantic validation
Phase D — Better preview
Phase E — LLM-generated unique world bootstrap

Then small integration/UI polish and tests

Do not start with preview first. Preview quality depends on having the richer setup fields and semantic validation in place.

Phase E — LLM-generated unique world bootstrap
Objective

The generated output must be:

bounded
editable

previewable before launch
optional, not mandatory
Core design rule

LLM generates proposals, not truth directly.

The LLM should never directly mutate canonical simulation state.

Correct flow:

Builder inputs are normalized
LLM generates structured proposal payload
Backend parses and validates it
Proposal is shown to user
User accepts/regenerates/edits
Accepted proposal is merged into canonical setup payload
Launch uses accepted structured data only

That is the safest version for your project.

E10 — Determinism / replay / launch safety

Because your architecture cares about determinism, generation must be treated carefully.

Best approach

Generation happens only during setup time, before launch, and accepted output is stored in canonical setup JSON.

That means runtime simulation does not need to regenerate anything.

So determinism is preserved by:

generating once
storing accepted structured output
launching from stored output only
Optional provenance metadata

Store in setup:

"generated_package_meta": {
  "used_llm": True,
  "generator_version": "v1",
  "generated_at": "...",
  "preferences": {...},
}

This helps debugging and later migrations.
