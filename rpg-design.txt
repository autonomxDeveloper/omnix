Goal

Transform the RPG from:

player input → one response

into:

persistent living world
NPCs act and talk without player initiation
world events occur while the player is idle
ambient narration and interruptions arrive out of turn
all of it remains deterministic, bounded, replay-safe, and server-authoritative
Non-negotiable architecture rules

The coding LLM must preserve these rules throughout all phases:

1. Server authoritative only
The frontend must never invent ambient events.
The frontend may only:
subscribe
request idle advancement
render authoritative payloads
2. Deterministic and replay-safe
Ambient updates must come from canonical session state.
No JS timer randomness may affect truth.
Any nondeterministic LLM output used for ambient text must remain presentation-only and be capture/replay safe.
3. Presentation must not mutate simulation truth
The sim decides what happened.
Narration/dialogue decides how it is expressed.
4. Bounded state only

Hard caps required for:

ambient queue
unread queue
pending interruptions
cooldown maps
catch-up summaries
recent delivered event ids
5. Fail hard on missing required replay data

If any ambient presentation path depends on capture data during replay:

no silent fallback
return structured failure
expose diagnostics
Current reality in this codebase

From the current uploaded tree:

src/app/rpg/session/runtime.py
canonical turn execution exists
world/sim updates are tied to player turn flow
no true idle tick product layer yet
src/app/rpg/api/rpg_session_routes.py
only one-shot turn routes exist
/api/rpg/session/turn/stream streams a single narration response, then closes
src/static/rpg/rpg.js
only per-turn streaming exists
no persistent session SSE
no ambient heartbeat loop
no unread/interruption model
src/app/rpg/creator/world_simulation.py
already has deterministic simulation stepping and NPC/world event machinery
this is the correct backbone for a living world
src/app/rpg/ai/world_scene_narrator.py
already does turn narration shaping
should be extended, not made authoritative

So the correct implementation is a runtime/session/product-layer expansion, not a rewrite of the simulation core.

What the coding LLM must not do

It must not:

let JS timers create world truth
let the narrator invent canonical world changes
let ambient speech mutate simulation state directly
create unbounded ambient history
create duplicate feed insertion on reconnect
silently fallback to live narration during replay if capture is required
Minimal viable ambient behaviors for first full ship

Even if the coding LLM does everything in one pass, the first live behaviors should be limited to these:

nearby NPC talks to player
nearby NPC talks to another nearby NPC
visible world event narration
important arrival/departure
encounter/escalation warning
companion commentary

That is enough to make the world feel alive.

Delivery format I’d ask the coding LLM for

Ask it to implement in this output structure:

Part 1

Runtime/session schema and idle tick engine

Part 2

Ambient builder + queue + policy

Part 3

Ambient dialogue + narrator ambient mode

Part 4

Session routes: idle tick, poll, persistent stream

Part 5

Frontend persistent subscription + heartbeat + ambient rendering

Part 6

Resume/catch-up + persistence/migration + replay safety

Part 7

Tests

That gives you a clean review sequence while still being “one go”.

Acceptance checklist for the finished implementation

You should be able to verify all of these manually:

start a session and do nothing for 15–20 seconds
ambient updates appear without player input
nearby NPCs can speak first
NPCs can speak to each other
player can receive a direct interruption
refresh page, no lost canonical ambient history
reconnect does not duplicate everything
idle for a while, return, receive a bounded summary/catch-up
replay/capture mode does not silently go live
feed remains readable and bounded
